### Caveats técnicos
- **Conflictos de naming**: `skill-optimizer` existe tanto en `hqhq1025/skill-optimizer` (1 skill, 16 installs) como en `mcollina/skills` (426 installs). Son implementaciones diferentes; `mcollina` tiene más tracción pero `hqhq1025` enfatiza análisis con datos reales de sesión. Recomendación: probar ambos y quedarse con el que mejor se ajuste al stack.
- **obra/superpowers vs obra/superpowers-skills**: Son repos SEPARADOS. `obra/superpowers` (14 skills, 921.7K installs) ya está instalado. `obra/superpowers-skills` (31 skills, 439 installs) es un repo adicional con skills de mindset/patrón (meta-pattern-recognition, inversion-exercise, collision-zone-thinking). No hay conflicto de paths si se instalan ambos; escriben en `.claude/skills/` con prefijos de repo.
- **Instalación de `alirezarezvani/claude-skills`**: NO está indexado en skills.sh. Requiere `/plugin marketplace add` de Claude Code o instalación manual (`git clone` + copy). No es compatible con `npx skills add`. Esto limita su portabilidad a otros agentes (Codex, Cursor) a menos que se use el script de conversión incluido.
- **Instalación de `ShunsukeHayashi/agent-skill-bus`**: Es un paquete npm (`agent-skill-bus`), no una skill en formato SKILL.md. Requiere `npx agent-skill-bus init` y modifica el proyecto con archivos JSONL (`.agent-skill-bus/`). Funciona como runtime complementario, no como skill cargada por Claude Code en contexto.
- **NeoLabHQ context-engineering-kit**: Tiene 133 skills en un solo repo. La instalación es monolítica (`npx skills add neolabhq/context-engineering-kit`). Esto carga 133 descripciones en startup (bajo costo token) pero puede generar ruido en triggering si las descripciones no son lo suficientemente específicas. Recomendación: instalar y monitorear trigger precision.
- **mattpocock/skills**: Alta tracción (365.3K installs). Algunas skills (`grill-me`, `caveman`) son opinionadas y pueden no alinearse con la cultura de MWT/FB. Recomendación: instalar selectivamente con `-s <skill>` en lugar del bundle completo.
- **garrytan/gstack**: 53 skills, 8.5K installs. Diseñado para estructurar un equipo de ingeniería virtual completo. Skills como `office-hours` y `design-review` son subjetivas; requieren project brief/context previo para buenos resultados. Recomendación: crear un `AGENTS.md` de contexto antes de usar skills opinionadas.
- **prompt-security/clawsec**: 13 skills, 2.4K installs. Algunas skills (`picoclaw-self-pen-testing`, `hermes-attestation-guardian`) tienen <15 installs y pueden ser inmaduras. Recomendación: priorizar `clawsec-suite`, `openclaw-audit-watchdog`, y `soul-guardian`.
- **hamelsmu/evals-skills**: 7 skills, 1.6K installs. Son herramientas de evaluación de LLM pipelines, no de "testing de agentes coding" propiamente. Útiles para FaberLoom que evaluará agentes virtuales de PYMEs, pero requieren adaptación para evaluar trazas de agentes (no solo outputs de LLM).
- **zscole/model-hierarchy-skill**: 59 installs, muy poco adoptado. Es experimental. El concepto (routing por complejidad de tarea) es valioso pero la implementación puede no estar probada en producción.
- **Vercel eval findings**: El blog de Vercel (enero 2026) demostró que `AGENTS.md` outperforms skills en evals (100% vs 53% baseline). Implicación: skills de workflow deben combinarse con `AGENTS.md` explícito para máxima fiabilidad, no confiar solo en triggering automático de skills.
- **Duplicados funcionales**: Hay ~8 skills de TDD/red-green-refactor en el ecosistema (obra, mattpocock, neolabhq, iliaal). Instalar múltiples puede causar conflicto de prioridad si dos skills dan instrucciones ligeramente diferentes. Recomendación: elegir UNO (sugerencia: `mattpocock/skills:tdd` por madurez) y descartar los demás.
- **Paths de escritura**: Todas las skills de skills.sh escriben en `.claude/skills/` (project-level) o `~/.claude/skills/` (user-level). `npx skills add` por default usa project-level. Para MWT con 430 archivos KB y FaberLoom multi-tenant, recomendación: usar project-level skills versionadas en Git para reproducibilidad.
