```markdown
## Reglas de diseno para SaaS B2B

### Color
- Usar maximo 1 color de acento por tenant
- Grayscale para UI chrome, accent solo para acciones primarias
- Nunca usar gradientes para fondos de componentes
- Opacidad de sombras: 4-8% para cards, 12-16% para modales

### Tipografia
- Ratio de headings: 1.25x (mobile) a 1.333x (desktop)
- Line-height: 1.5 para body, 1.2 para headings
- Font stack: system-ui primero, luego sans-serif

### Layout
- Spacing base: 4px (0.25rem)
- Max-width para contenido: 1200px
- Card padding: 24px (1.5rem)
- Gap entre elementos relacionados: 16px (1rem)

### Componentes
- Border-radius: 8px para cards, 4px para inputs, 9999px para pills
- Bordes: 1px solid con opacidad 10-15% del color de texto
- Transiciones: 150ms ease para estados interactivos
```

## Anti-patterns observados en skills publicas que NO replicar

### 1. Skills con "AI slop" como unico guidance
Muchos skills genericos dicen "evita AI slop" sin dar direccion concreta. El skill `frontend-design` de Anthropic es la excepcion que da direccion visual clara. **No replicar:** skills vagos sin ejemplos concretos.

### 2. Copiar tokens de marcas sin adaptacion
El repo `awesome-design-md` tiene tokens de marcas reales (Stripe, Vercel, etc.). Copiarlos tal cual es infringement de identidad visual. **No replicar:** uso directo de tokens de terceros sin adaptacion.

### 3. Skills que no validan output
Varios skills de design system generan codigo sin paso de validacion. StyleSeed es mejor porque tiene `/ui-review` y `/ui-audit`. **No replicar:** generacion sin validacion de consistencia ni a11y.

### 4. Dependencia de prompts leaked (v0, Bolt, Lovable)
Los system prompts leaked son referencia de como funcionan los servicios, pero copiarlos es: (a) legalmente gris, (b) tecnicamente fragil, (c) no aplicable a FaberLoom que es B2B multi-tenant. **No replicar:** prompts de servicios cerrados.

### 5. Skills GPL/AGPL en el pipeline
Algunas herramientas de diseno (p ej, componentes especificos) usan GPL. En un producto comercial cerrado esto es riesgo viral. **No replicar:** cualquier componente o skill con GPL/AGPL.

### 6. Hardcodear estilos en vez de usar tokens
Muchos skills generan CSS inline o Tailwind arbitrario sin referencia a tokens. Esto rompe multi-tenancy. **No replicar:** estilos hardcodeados que ignoren el design system del tenant.

### 7. Skills que no distinguen mobile/desktop
Algunos skills asumen desktop-only. Para SaaS B2B moderno, responsive es obligatorio. **No replicar:** skills sin consideracion responsive.

## Gap analysis: Que NO se encontro publicamente y FaberLoom debe construir desde cero

### 1. Multi-tenant design token resolution
Ningun skill o sistema publico maneja tokens por tenant con fallback a defaults. Style Dictionary no tiene concepto de tenant. **FaberLoom debe construir:**
- Token resolver que mergea `base.tokens.json` + `{tenant}.tokens.json`
- CSS Custom Properties con scoping por tenant
- Cache de tokens compilados por tenant

### 2. Brand-as-code pipeline completo
Aunque existen DESIGN.md y DTCG por separado, no hay un pipeline integrado que: (1) tome un DESIGN.md, (2) genere tokens DTCG, (3) compile a CSS, (4) valide contra reglas, (5) sirva al agente. **FaberLoom debe construir:**
- Pipeline `design.md -> dtcg -> css -> component-registry`
- Validacion de consistencia entre DESIGN.md y tokens generados
- Hot-reload de tokens en dev mode

### 3. Skill de agente para generacion de mocks B2B
Ningun skill publico esta optimizado para "generar un dashboard de analytics B2B" o "generar un form de configuracion de tenant". Los skills existentes son genericos (frontend-design) o consumer-focused (StyleSeed). **FaberLoom debe construir:**
- `faberloom-designer` skill especializado en patrones B2B (dashboards, tables, forms, settings, onboarding)
- Templates de mock por tipo de pantalla B2B
- Reglas de consistencia con FaberLoom's component registry

### 4. Sistema de component registry agent-visible
Ningun proyecto publico expone su component registry de forma que un agente pueda descubrir y usar componentes automaticamente. **FaberLoom debe construir:**
- Component registry con metadata (props, variants, tokens usados, ejemplos)
- MCP server para que agentes descubran componentes
- Integration con el skill para auto-seleccion de componentes

### 5. Cross-session design memory
Los skills actuales son stateless. Para FaberLoom, el agente deberia recordar decisiones de diseno previas por tenant. **FaberLoom debe construir:**
- Design memory per-tenant (decisiones previas, feedback, iteraciones)
- Versionado de design decisions
- Sincronizacion entre sesiones
